# 壁効果モーメントと制御トルクのタイミング

## 調査結果：ズレの原因

- **制御トルク**：各**制御ステップ**の先頭で、その時点の状態 `st` から PD + FF を計算し、ミキサで `omega2_cmd` を出して、その制御ステップ内の**全物理サブステップ**で同じ `omega2_cmd` をモータに与えている。
- **壁効果モーメント**：もともと制御ステップ先頭で一度だけ `wall_active`, `tau_wall_b` を計算し、同じ値をその制御ステップ内の全サブステップで外力として印加していた。

したがって次のようなズレが発生していた。

1. **サブステップ数 > 1 のとき**  
   1 制御ステップで複数回物理積分するため、ステップ途中で機体が「壁ゾーン入り/出し」しても、壁トルクはステップ先頭の位置で計算した値のまま。  
   → 実際に壁に差し掛かっているタイミングと、壁効果モーメントを印加するタイミングが最大で **1 制御周期** ずれる。

2. **サブステップ数 = 1 のとき（例: hz=physics_hz=240）**  
   制御と物理が 1:1 なので、印加する壁トルクも「1 ステップ前の状態」に基づく。  
   → 境界をまたぐ瞬間は「入った次のステップから壁トルク」「出た次のステップまで壁トルク」となり、最大 **1 物理ステップ** の遅れが残る。

3. **フィードフォワード**  
   FF は `tau_ff = -tau_wall_b` を制御ステップ先頭で一度だけ計算してミキサに含めている。  
   → 壁トルクを「今の位置」で毎サブステップ更新するようにしても、FF は「ステップ先頭の壁トルク」で固定のため、サブステップ数 > 1 のときは **制御トルクと壁効果のタイミングにずれ** が残る。

## 実施した修正

- **壁効果モーメントの印加タイミング**を「制御ステップ先頭」から「各物理サブステップの先頭」に変更した。
  - 共通の `_compute_wall_torque(st, args)` を用意し、制御ステップ先頭では `st`、サブステップ内ではそのステップ開始時の `env.get_state()` を渡す。
  - 制御用（FF・ミキサ）には従来どおり「制御ステップ先頭の状態」で計算した `tau_wall_b` を使用。
  - **外力としての壁トルク**は、各物理サブステップで「その時点の状態」から `_compute_wall_torque` を呼び、得た `tau_wall_b_sub` をワールド座標に変換して `apply_body_wrench_world` で印加する。

これにより、

- 壁ゾーンに入った/出た瞬間に、**壁効果モーメントの印加**は「今の位置」に一致する。
- ログ・CSV 用の `wall_active` / `d_wall` は従来どおり制御レート（ステップ先頭の状態）で計算した値を使用。

## 残るずれ（オプション改善案）

- **FF のタイミング**：サブステップ数 > 1 のとき、FF はあくまで「制御ステップ先頭の壁トルク」で打っているため、サブステップ後半では実機の壁トルクと FF に少しずれが残る。  
  完全に揃えるには、**制御そのものを物理レートで実行**（各サブステップで状態を取得 → 軌道・壁・PD+FF・ミキサを計算 → 推力・壁トルクを印加）する必要がある。
